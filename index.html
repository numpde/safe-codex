safe-codex

Run Codex in a disposable Podman or Docker container with a small, explicit host surface.

What It Does

Small host surface

Only the selected workspace and explicit extra mounts are exposed to the container.

Disposable runtime

The container root filesystem is read-only, capabilities are dropped, and containers are removed on exit.

Separate Codex state

Container Codex state lives in an engine named volume, not host ~/.codex.

Regulated network

Default egress goes through Squid, with local/private destinations blocked.

Rootless Podman is preferred when available. Docker works, but a user in the docker group should be treated as effectively root-capable on the host.

The launcher parses .env as plain KEY=VALUE. It does not source shell code, and it accepts only SAFE_CODEX_* keys plus OPENAI_API_KEY.

Build

Check what the launcher sees before building or running:

./bin/safe-codex-podman doctor

Build with Podman:

./bin/safe-codex-podman build

Build with Docker:

./bin/safe-codex-docker build

The build uses npm once inside the build container with --ignore-scripts, then removes npm, npx, and corepack from the runtime image.

Run Codex

From the project you want Codex to work on:

/path/to/safe-codex/bin/safe-codex-podman run

Docker equivalent:

/path/to/safe-codex/bin/safe-codex-docker run

Default behavior

  • Profile: default.
  • Workspace: current directory mounted at /workspace read/write.
  • Persistent state: safe-codex-state-default mounted at /home/codex/.codex.
  • Nonpersistent home: /home/codex tmpfs with noexec.
  • Network: proxy, using safe-codex-proxy-nonlocal.
  • Lifecycle: Codex containers are removed on exit; state persists in the named volume.

For a read-only-by-default workflow, set SAFE_CODEX_WORKSPACE_MODE=ro in .env, then pass --workspace-rw only when a run should edit the workspace.

Detach and resume

Detach from a live container without stopping Codex:

Ctrl-p Ctrl-q

Attach again:

/path/to/safe-codex/bin/safe-codex-podman attach

If the machine restarted or Codex exited, run the same run command again. The process is gone, but Codex auth/session files persist in the profile volume.

Ask Codex to resume from the persistent volume:

/path/to/safe-codex/bin/safe-codex-podman run -- resume

Storage Model

The container home is split deliberately.

Container path Backing storage Persistence
/home/codex/.codexEngine named volume safe-codex-state-PROFILEPersists
/home/codex/.cachetmpfs under /home/codexDeleted on exit
/home/codex/.localtmpfs under /home/codexDeleted on exit
/home/codex except .codextmpfs with nosuid,nodev,noexecDeleted on exit
/tmptmpfs with nosuid,nodev,noexecDeleted on exit
/runtmpfs with nosuid,nodev,noexecDeleted on exit
/workspaceSelected host workspace bind mountHost files

For the default Podman profile, the persistent volume is stored on the host at:

~/.local/share/containers/storage/volumes/safe-codex-state-default/_data

Inspect through Podman so rootless user-namespace ownership is displayed correctly:

podman volume inspect safe-codex-state-default
podman unshare ls -la ~/.local/share/containers/storage/volumes/safe-codex-state-default/_data

Do not mount or copy host ~/.codex into the container. The profile volume may contain Codex auth material, logs, sessions, plugins, skills, and local state.

Profiles

Use separate profiles for separate trust zones:

/path/to/safe-codex/bin/safe-codex-podman run \
  --profile id-overlay \
  --workspace /path/to/id-overlay

This creates container name safe-codex-id-overlay and state volume safe-codex-state-id-overlay. Profile names may contain only letters, numbers, dot, underscore, and dash.

List existing persistent profiles and any live Codex containers:

/path/to/safe-codex/bin/safe-codex-podman profiles

Multiple Sessions

A profile controls Codex's persistent state. A workspace controls which host files are mounted at /workspace. Changing the profile while keeping the same workspace gives you a separate Codex state volume looking at the same host files.

/path/to/safe-codex/bin/safe-codex-podman run \
  --profile edit-1 \
  --workspace /path/to/project \
  --workspace-rw

/path/to/safe-codex/bin/safe-codex-podman run \
  --profile edit-2 \
  --workspace /path/to/project \
  --workspace-rw
Setting What changes What is shared
--profile edit-1 versus --profile edit-2 Separate containers and separate state volumes: safe-codex-state-edit-1 and safe-codex-state-edit-2. Nothing in ~/.codex is shared between those profiles.
Same --workspace /path/to/project The same host directory is mounted into each container at /workspace. Both sessions see the same files, git state, and uncommitted changes in that host directory.

If two sessions mount the same workspace read/write, they can edit the same files at the same time. The profile boundary separates Codex state; it does not isolate the shared workspace bind mount.

A safer pattern is one writable editor plus one or more read-only reviewers:

/path/to/safe-codex/bin/safe-codex-podman run \
  --profile edit \
  --workspace /path/to/project \
  --workspace-rw

/path/to/safe-codex/bin/safe-codex-podman run \
  --profile review \
  --workspace /path/to/project \
  --workspace-ro

If two sessions should edit independently, use separate git worktrees or clones and give each one its own profile.

Read-Only and Extra Mounts

Inspect a repo read-only:

/path/to/safe-codex/bin/safe-codex-podman run \
  --workspace /path/to/suspicious \
  --workspace-ro

Override a read-only .env default for a normal editing session:

/path/to/safe-codex/bin/safe-codex-podman run \
  --workspace /path/to/project \
  --workspace-rw

Add one writable output folder:

/path/to/safe-codex/bin/safe-codex-podman run \
  --workspace /path/to/project \
  --rw /tmp/codex-output:/output

Add a read-only reference folder:

/path/to/safe-codex/bin/safe-codex-podman run \
  --workspace /path/to/project \
  --ro /path/to/docs:/mnt/docs

The launcher refuses risky mounts by default, including /run, /var/run, /dev, /proc, /sys, ~/.ssh, ~/.codex, and mount trees containing Unix sockets. Symlinks are resolved before this check. Paths containing commas or newlines are rejected because Docker and Podman --mount cannot represent them unambiguously.

Extra mount destinations are constrained to:

/mnt/rw...
/mnt/ro...
/workspace/...
/input...
/output...

Override these checks only with a concrete reason:

--allow-risky-mounts

Network Modes

Mode Behavior
--network proxy Default. Codex uses Squid on an internal network. The proxy blocks local/private destinations such as loopback, RFC1918 ranges, link-local, multicast, and IPv6 local ranges.
--network openai Stricter allowlist for OpenAI/ChatGPT-related domains. Useful for API-key use; often too strict for browser login, package registries, git remotes, or MCP servers.
--network none No network. Useful for offline inspection; Codex API calls will not work.
--network bridge --allow-bridge-network Unrestricted container bridge. This bypasses the proxy and is intentionally gated.

The proxy uses a dedicated egress network named safe-codex-egress-nonlocal unless SAFE_CODEX_EXTERNAL_NETWORK is set.

Authentication

Preferred: pass an API key from the environment.

export OPENAI_API_KEY=...
/path/to/safe-codex/bin/safe-codex-podman run

Do not mount host ~/.codex. That directory contains live auth/session material. If you use browser/login auth inside the container, it is stored in the profile's named Codex-state volume.

Environment variables passed with -e or --env-file are visible through container inspection while the container is running. The launcher removes the Codex container on exit so those values are not left in stopped container metadata.

Shell

Open a shell with the same containment settings:

/path/to/safe-codex/bin/safe-codex-podman shell \
  --workspace /path/to/project

Maintenance

Show readiness:

./bin/safe-codex-podman doctor

Show containers, volumes, and networks:

./bin/safe-codex-podman status

List persistent Codex profiles:

./bin/safe-codex-podman profiles

Inspect a specific proxy profile:

./bin/safe-codex-podman proxy status --mode openai

Stop a live Codex container:

./bin/safe-codex-podman stop --profile default

Remove a running container but keep persistent Codex state:

./bin/safe-codex-podman rm --profile default

Delete a profile's persistent Codex state only when intended:

podman volume rm safe-codex-state-default

Checks

Run local syntax checks without building images:

./bin/check

If safe-codex-proxy:latest already exists, this also runs Squid's parser against both proxy profiles. If shellcheck is installed, it is used too.
